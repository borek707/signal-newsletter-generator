# Signal Sources

Primary sources for discovering trending discussions in distributed systems.

## Reddit Communities

### r/scala (scala.reddit.com)
- **Focus**: Scala ecosystem, functional programming, JVM performance
- **Signal types**: Library releases, migration stories, performance comparisons
- **Good patterns**: "We migrated from X to Scala", "Why we chose Scala for..."

### r/rust (rust.reddit.com)
- **Focus**: Systems programming, memory safety, performance
- **Signal types**: Rewrites, benchmarks, adoption stories
- **Good patterns**: "Rewrote X in Rust", "Rust vs Go for...", "Memory safety in production"

### r/apachekafka (apachekafka.reddit.com)
- **Focus**: Streaming infrastructure, Kafka operations
- **Signal types**: Migration stories, cost optimization, operational issues
- **Good patterns**: "Migrating from Kafka to...", "Kafka at scale", "Kafka latency issues"

### r/dataengineering (dataengineering.reddit.com)
- **Focus**: Data pipelines, infrastructure decisions
- **Signal types**: Tool comparisons, architecture decisions, cost discussions
- **Good patterns**: "Why we switched from X to Y", "Data pipeline at scale"

## Engagement Thresholds

Look for posts with:
- 50+ upvotes
- 30+ comments
- Posted within last 30 days
- Discussion in comments (not just link sharing)

## Signal Quality Indicators

Good signals contain:
- **Specific metrics**: "20x improvement", "50% cost reduction"
- **Trade-off discussion**: "We lost X but gained Y"
- **Production experience**: Real incidents, real decisions
- **Technical depth**: Not just announcements

## Signal Types to Prioritize

1. **Migrations**: "We moved from X to Y"
2. **Benchmarks**: Performance comparisons with methodology
3. **Incidents**: Production issues and resolutions
4. **Cost discussions**: Infrastructure economics
5. **Architecture decisions**: "Why we chose X"

## Signal Types to Avoid

- Vendor announcements
- Marketing content
- Tutorial posts
- "Show HN" style launches
- Opinion pieces without data
