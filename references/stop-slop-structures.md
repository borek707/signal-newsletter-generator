# Stop-Slop Reference: Structural Patterns to Avoid

AI writing follows predictable structural patterns. Break them.

## Binary Contrasts

Never write "not X, it's Y" patterns. State Y directly.

| Avoid | Use Instead |
|-------|-------------|
| It's not about cost, it's about value | Value matters more than cost |
| This isn't a technical problem, it's a people problem | People cause this problem |
| Not only faster, but also cheaper | Faster and cheaper |
| Don't focus on features, focus on outcomes | Focus on outcomes |

## Negative Listings

Never list what something is not. List what it is.

| Avoid | Use Instead |
|-------|-------------|
| This isn't your typical migration | This migration required three phases |
| Not just another tool | A specialized tool for |
| This isn't about | This addresses |

## Dramatic Fragmentation

Avoid staccato sentences that artificially create drama:

| Avoid | Use Instead |
|-------|-------------|
| The results were clear. Very clear. Too clear. | The results surprised everyone. |
| One problem. One solution. One team. | The team solved one problem. |

## Rhetorical Setups

Never ask questions you immediately answer:

| Avoid | Use Instead |
|-------|-------------|
| Why does this matter? Because | This matters because |
| What's the solution? Simple: | The solution: |
| How did they do it? By | They did it by |

## False Agency

Inanimate objects cannot perform human actions:

| Avoid | Use Instead |
|-------|-------------|
| The complaint became a fix | The team turned the complaint into a fix |
| The error taught us | We learned from the error |
| The decision emerged from | The team decided |
| The architecture allows for | Engineers can use the architecture to |

## Narrator-from-a-Distance

Put the reader in the room, not observing from afar:

| Avoid | Use Instead |
|-------|-------------|
| Nobody designed this | You wouldn't design this |
| People often ask | You might ask |
| Engineers typically | Your team probably |
| Organizations struggle | Your company might struggle |

## Passive Voice Patterns

Always find the actor and make them the subject:

| Passive | Active |
|---------|--------|
| The migration was completed | The team completed the migration |
| Errors were discovered | The monitoring system caught errors |
| A decision was made | Leadership decided |
| Tests were run | QA ran tests |
| The system was designed | The team designed the system |

## Three-Item Rhythm

Vary your lists. Two items often work better than three:

| Avoid | Use Instead |
|-------|-------------|
| Faster, cheaper, better | Faster and cheaper |
| Simple, elegant, powerful | Simple and powerful |

## Punchy Endings

End paragraphs differently. Never end with a one-liner designed to resonate:

| Avoid | Use Instead |
|-------|-------------|
| And that made all the difference. | The team shipped two weeks early. |
| Sometimes the answer is simple. | The fix required one config change. |
| The rest is history. | They migrated 50 services in six months. |

## Em-Dash Usage

Never use em-dashes. They are AI-typical.

| Avoid | Use Instead |
|-------|-------------|
| The cost savings — substantial but delayed — appeared | The cost savings appeared. They were substantial but delayed. |
| One thing mattered: speed | Speed mattered most. |
| The solution — a complete rewrite | The solution required a complete rewrite. |
